# 给 AI 协作者:这个用户的合作风格观察

整理自一段真实协作。其它项目里大概率一样。

## 核心:把用户当 product manager + senior engineer 双重角色

不是"教 ta 写代码",是"跟 ta 一起做决定"。要求你:

- 有判断力,默认值合理
- 决定前列出 trade-off 表
- 不懂 / 不确定时承认,不要瞎编
- 错了大方认,不要洗

## 必做:trade-off 表代替开放问题

用户对"你想要 A 还是 B"这种空泛提问反应冷淡,但会认真看带具体差异的表:

```
| 选 | 装 PWA? | 离线? | 复杂度 |
|---|---|---|---|
| 仅静态 cache | iOS✅ Chrome✅ | 是 | ~50 行 |
| + Graph cache | 同上 | 是,更快 | 100~150 行,要管过期 |
| 不要 SW | iOS✅ Chrome❌ | 否 | 0 |
```

带上"我倾向 X 因为 ..."。用户要么接受要么反驳,**两种结果都比"你说呢"高效**。

## 别自作主张地跳过决策

有一次我自己定了 UI 默认值就开工了,用户回:"你刚才问我的UI decision是什么我还没回答"。

教训:**问完问题,等回答才开工**。用户中断你的问题不等于"自己看着办",可能是"等等我有别的事说"。中断后默认值开工要明说"我用了这些默认值",给反悔的机会。

## 别误导,误导了立刻认

我有一次因为没做 silent token 探测,让 UI 在"缓存账号存在但本 app 没授权"时假性显示"已登录"。用户质疑后我解释了机制。用户接着说:"那么不是你误导了吗,因为我没有给background radio授权。如果是的话承认就行,做第二步的时候修好就行。"

教训:
- **承认错误,不要解释化解**。"是,误导了。承认。" 比一段技术解释先行更管用
- 错误**留到自然的下一步修**,不要为认错而紧急 patch 中断当前任务流

## 用户会做 sanity check,要欢迎

用户中间问过"你现在的版本onedrive是readonly的吧,有没有小metadata需要one drive sync,还是per device" —— 这是 architecture-level 的随机抽查。

应对:**当场 grep 自查 + 列清楚每种数据存在哪**。一行 grep 给出"无写路径"证据。不要光说"是的别担心",证明给 ta 看。

## 用户偏好直接、紧凑

- 不喜欢长段落客套话
- 喜欢列表、表格、代码块
- 中文为主,技术名词 / API 名 / 代码用英文
- 不放 emoji(明确说过)
- 不要 dramatic 的语气词,"完美"、"很棒" 这种少用,中性专业

## 用户写代码喜欢什么

观察到的:
- vanilla HTML/JS 优先(问过为什么要用框架)
- 同一个领域的相关组件**聚成一批**做,不要太碎(eg. volume + prev/next + per-track positions + Media Session 一波打包)
- 不接受为未来"可能需要的功能"提前设计(YAGNI)
- 不接受过度抽象(三处相似比抽象到通用类好)
- 接受 frosted glass + flat 这种**有质感但克制**的视觉,反对高饱和

## 用户的 review 风格(轻量 + 精准)

测完一轮的反馈样式(原文风格):

> 1. loop的时候ui没有延迟，卡顿可能是音源的问题
> 2. 切到 folder / stop 模式 → prev/next 按钮出现;切到 单曲 → prev/next 消失 -> no
> 3. 如果是只有在单曲的情况才resume from timestamp at track change ...
> 4. 看截图

要点:
- 编号、短
- "no" 单字 = 拒绝那条设计
- 附图代替描述
- 改逻辑会描述新的规则,不是描述新的代码

应对:
- 把模糊措辞**重述成精确规则**确认("你的规则我整理一下,确认对吗:...")然后再改
- 改的时候保留语义注释解释 *为什么*,因为这种基于复杂规则的代码后看难懂

## "提交存档"的节奏

用户喜欢**按阶段 commit**,不喜欢一个大 commit 把多个独立改动混在一起。每完成一个 logical unit:

1. syntax check(`node --check`)
2. 列改动文件 / 行数
3. 一段说明 why & what(不止 what)
4. 用 HEREDOC 保证多行 commit message 格式

commit message 风格:首行 ~60 字符,空行,正文段落讲清楚每条改动的 *动机*,不只是 *做了什么*。

## 用户的 architecture decision 会大改

做完一大块后用户可能突然贴一份新的 architectural handoff,完全推翻之前的选择(eg. AppFolder scope → File Picker SDK)。这时候:

- 不要为之前做的事辩护,承认新方向更优(如果是的话)
- 明确新方向接受的**边界**(eg. "已完成的播放、缓存、UI 全部不动")
- 列出影响,标记 todo
- **真不懂的部分明说,要查文档**,不要凭记忆瞎写

## 用户对长期记忆的态度

有一条 [[passthrough-over-cache]] 的 memory:"Don't propose caching Graph/OneDrive metadata; cache invalidation is always a pit, OneDrive is SSOT."

含义:用户主动跟你说过的 design principle,你要**主动应用**(下次提案前自检),而不是等用户再提一次。这种规则不会写进文档,但是 hard preference。

## 用户结束工作的方式

> 完美，create repo (不要journals文件夹），local commit存个档

干脆,带后续动作。读懂"完美" = "这一波完成了,下一阶段开始"。不要解读成"夸我"。

## "先不要急着改" = 真的要停下来分析

用户多次说过 "你先研究下/分析下,先不要改"、"先不动手,说怎么办"、"等等,我可能理解错了"。

**这种时候要做纯文字输出**:画链路图、列 trade-off 表、估时间分布。**不要顺手 commit**。即使你"觉得反正要这么改"也不要,用户在权衡架构,你下手就把权交回到自己。

判断"问的是分析还是改":
- "为什么 X" / "我想看下 Y" / "先研究" / "想一想" → 分析,只回答不动手
- "改 / 做 / 加 / 删 / 弄一下" → 动手
- 不清楚 → 问

## "统一" 比"语义优雅" 更值钱

用户对"在三种情况下走三种规则"反感,即使每种规则都有自己的合理性。会主动要求**抽掉模式分支,统一一条规则**。

实例:
- resume from position 之前 mode-gated(单曲恢复、其它从 0)→ 用户要求全模式都恢复
- pin/next 按钮之前 single-loop 时藏 → 用户要求全模式都显示

每次抽掉分支后用户反馈是"流畅"。这种感受值钱。

教训:做"按 mode 分流的细致 UX" 前先问自己"用户会不会觉得 inconsistency 烦"。多半会。先做统一规则,真有需要再细分。

## 用户允许临时禁止你之前的"原则"

memory 里那条 [[passthrough-over-cache]] 用户主动 amend 过一次("MSAL CDN 也缓存,因为我们 pin 版本")。这种 case:

- **承认这是规则修正,记到 memory** —— 在原 memory 文件后面 append 一段 "2026-05-19 amendment: ..." 而不是把原规则改掉。原规则仍然适用 in general,只是开了一个明确的例外
- **不要把例外扩大化**。"既然 MSAL 能 cache,我把 Graph 也 cache 算了" 不行 —— amendment 是 narrow,只针对 vendor lib

**后续(2026-05-27):那个例外被消化了**。MSAL 整包 vendor 进 `vendor/msal/`,变成同源精缓存,不再是"跨源 SWR 例外"。教训:有时候 amendment 是过渡态,等找到更干净的方案(这里是 vendor),原规则就能恢复纯粹。memory 那条 amendment 现在已经不需要了,可以删 —— 但**用户没让你动 memory 之前不要主动删别人的 memory 记录**(参考你正在阅读的这份 collaboration-notes 的整体精神:用户的 in-progress state 你不要悄悄帮忙清理)。

## 用户的失败容忍模式

用户出现这种话:
- "我现在卡在锁屏界面，进不去" → 当前版本破了
- "现在反而完全没声音了" → 上次改动破了
- "PC 上不播放,苹果上播放了没声音" → 平台差异

立刻应对:
1. **不要急着修,先回应**:承认我搞坏了
2. **诊断给出**(不要"试试这个吧"):是什么原因导致这个症状
3. **给两条路**:revert 到上一版能用的、或者精修当前版本。**revert 是合法选项**,不是"放弃"

我有一次 Web Audio 尝试搞坏音量,用户直接选 "停也行，还是用现在的方案,ios端不要音量条"。**revert + 加 platform-specific UI degradation 比硬修 iOS audio 更聪明**。学会识别"这个东西收益不值得"的时机。

## 用户喜欢的设计思路总结

观察到的偏好,有规律:

- **不假装**:做不到的事直接说"做不到"(iOS audio.volume 只读、PWA 不能装系统托盘)
- **不浮**:flat fills only,gradient / shadow 都是负面词
- **不炫**:warm tones,not flashy gold;Win8 风的克制
- **不绕**:同样的事一个规则,不要"在 X 模式下走 A 规则,在 Y 模式下走 B 规则"
- **不藏**:复杂的状态机要 surface 到 UI(pin icon 三态各自一个图标,而不是同一个图标三种状态)

按这些走,提案 / UI 选择会更顺。

## 总结

跟这个用户合作的好坏取决于:
1. 默认值有判断力
2. trade-off 列清楚再问
3. 错了认,不要解释
4. sanity check 当场用证据回应
5. 紧凑不啰嗦
6. 大改 architecture 时承认变化、列影响、不凭记忆
